### Lessons learned (iteration 16)

- **Self-review pre-commit catches race conditions unit tests can't**: Three different self-review
  passes (code-reviewer + security-reviewer + architect) caught 1 CRITICAL TOCTOU UAF on
  `activeStreamHandle` (commit `f5d716a`'s round-3 review), 2 HIGH races on `current` field
  state-mutation in M6-CO1 (commit `dcce36f`), and 1 hidden self-cancel cascade that broke the
  ENTIRE M6-CO1 feature in production but was masked by the test API key always returning 401
  in ~600 ms (commit `c1dc07a`). Unit tests don't reproduce these without adversarial
  multithreading harnesses; multi-agent reviews caught them by inspection. Worth keeping in the
  M7+ playbook.

- **Diagnostic instrumentation > attempted fix when upstream-blocked**: M4-S14 emoji color
  rendering looked like a bug-to-fix until per-glyph swash content logging (commit `452e3c5`)
  proved swash 0.1.x simply doesn't decode COLR v1 (the modern Noto Color Emoji format on
  Android 15). The fix is upstream (cosmic-text fork → swash 0.2). Documenting 4 alternative
  paths in the result JSON saves any future contributor from repeating the investigation. The
  log lines are also kept in for production diagnostics behind log level Debug.

- **`debounceJob?.cancel()` inside `cancelActiveStream()` was a self-cancel trap**: Pre-existing
  bug in M6-CO1 round-1: `fireSuggestion` defensively called `cancelActiveStream` at its top to
  release any leftover stream, but `cancelActiveStream` ends with `debounceJob?.cancel()` —

  The 401 from the test key arrived in ~600 ms (after the orphan tokio task) — which is what

  shared state, trace the side effects all the way down.

- **API consumer surface boundaries matter for v1-prep**: The `Block.output: Vec<u8>` capture

  (64 KB) without worrying about partial multi-byte UTF-8 splitting. JSON dump does
  `String::from_utf8_lossy` at serialization time. Pattern: capture raw, serialize lossy at
  the boundary.

- **AGP signingConfig graceful degradation**: The build.gradle pattern
  `if (hasKeystore) { signingConfig signingConfigs.release }` lets the same source tree build
  signed APKs locally (with `keystore.properties`) AND unsigned APKs in CI / F-Droid (without).
  No conditional compilation, no fork. The keystore.properties file is gitignored; the
  generation instructions live as comments in the same build.gradle block where the config
  reads from it. This is the "secret-where-it's-used" pattern — anyone touching signing
  reads the docs by being in the right file.

### Lessons learned (iteration 17)

- **CI-as-discovery-mechanism**: 4 separate "CI failed" cycles each surfaced a real bug:
  warp-src clone path (sibling vs in-repo), deprecated actions-rs, NDK install ordering,
  cross-platform errno (`__error` is macOS-only). Without CI, these would have hit the next
  contributor's setup or the F-Droid build. CI green is the v1-ship gate, but CI red is the
  v1-correctness signal.

- **Self-cancel cascades hide behind transport errors**: M6-CO1 round-1 had a CRITICAL
  defect where `cancelActiveStream()` ended with `debounceJob?.cancel()`, cancelling the
  very coroutine running fireSuggestion. The CancellationException freed the handle within
  ~1ms of starting — the stream NEVER reached :DONE: under any condition. Hidden because
  the test API key always 401'd in ~600ms and the orphan tokio task surfaced that 401 even
  after Kotlin had freed the handle. Lesson: when validating with intentionally-failing
  inputs, always also test ONE successful path (or simulate one) to differentiate
  "transport says fail" from "client logic broken — never reaches :DONE:".

- **Strip vs LTO is two separate doors**: We removed 2.4 MB via strip (debug symbols)
  and another 1.4 MB via LTO + cgu=1 + panic=abort. They're additive — the strip removes
  what's there; LTO removes what shouldn't have been there in the first place (dead code

  whichever font on the device that swash 0.1.x can decode. Free. Diagnosis (per-glyph
  swash content logging at log::debug) was 80% of the work; the actual code change was

  pin-vs-upstream check feels inconvenient when upstream drifts (you have to

  silent upstream change would mean two builds of the same git SHA produce different

  artifact for every milestone close — it's the regression baseline.

- **Pre-built binaries beat cargo install for CI**: `cargo install --locked cargo-audit
  cargo-deny` adds 5+ minutes to every CI cycle. `taiki-e/install-action@v2 with: tool:
  cargo-audit,cargo-deny` does the same in ~5-10s. The trade is "build it ourselves" vs
  "trust the tool author's CI" — for v1-prep where speed matters, the latter is the right
  call. Re-evaluate if either tool starts shipping malicious binaries (we'll know via
  the security advisories the tools themselves catch).

### What we learned about Blocker #3

It was not zsh dying. zsh never started — `execve` returned errno 13
(EACCES) because of Android's SELinux policy. The bundled
`$PREFIX/bin/zsh` is labelled `app_data_file:s0`; the app process
runs in `untrusted_app:s0` domain; AOSP `system/sepolicy/private/
untrusted_app.te` since API 29 carries `neverallow untrusted_app
app_data_file:file execute`. The mount itself isn't `noexec`
(F2FS shows `nosuid,nodev` only on /data/user/0), but SELinux denies
the transition before it gets that far. `run-as` works because that
shell runs in a different domain that has the exception.

This is the same restriction Termux historically dodged via
`targetSdkVersion=28`. For v1.0 we ship around it with the
`/system/bin/sh` fallback. The structural fix is to load Termux
binaries from `nativeLibraryDir` (`system_lib_file:s0`-labelled,
exec-allowed) — that's an M4-shaped refactor scheduled for v1.1.

### What this means for v1.0 capabilities

The launcher path produces a working mksh shell with the
AccessoryRow + Gboard. mksh built-ins (`echo`, `cd`, `pwd`, `set`,
redirects, pipes) work; any binary in `$PREFIX/bin/` (zsh, ls, cat,
head, tail, uname, date, id, …) returns `Permission denied (errno
13)` because of the same SELinux domain restriction. The CI/release/
license/APK-shrink work all stays valid; we ship the v1.0 with this
clearly documented and the structural fix slated for v1.1.

### Lessons

- **"Shell dies in PTY" was a misframing**. The symptom (read loop
  ends ~10 ms after spawn) led the prior round to focus on PTY setup
  (TIOCSWINSZ, setpgid, tcsetpgrp). The real failure was earlier —
  `execve` itself. Adding a single stderr-printing diagnostic in the
  post-fork-pre-exec child path collapsed 4+ rounds of speculation
  into one log line. Lesson: when a child process exits before
  producing expected output, log the EXEC failure path explicitly,
  not just the read-side error.
- **Test the actual entry point real users will take**. Every prior
  M1-M6 device test launched via `am start ... --ez terminal_mode
  true` because that was the M3-S04 driver convention. The plain
  launcher Intent (no extras) is what real users see, and it had
  never been exercised end-to-end. Lesson: include a "tap the icon
  cold" pass in every milestone close, even when driver-shaped
  tests are passing.
- **SELinux is upstream of `mount noexec`**. The /data/user/0 mount
  doesn't carry `noexec`, so `mount` output suggested exec should
  work. SELinux denies it earlier. Lesson: when EACCES appears on
  `execve` in an app-private path, check the policy domain before
  the mount flags.
- **Renderer insets stored but never read = silent UI bug**. The
  `setRenderInsets` JNI accepted top/left/right/bottom values from
  the WindowInsetsCompat callback and stored them in atomics that
  no later code path consumed. The grid rendered into the full
  surface, sliding under the status bar. Defaulting fullscreen
  works around it for v1.0; the proper fix is making the renderer
  honor the top inset.

## Iteration 19 — IME → PTY input wiring (2026-05-02, commit c6a7359)

  - commitText(text) → text bytes
  - KEYCODE_ENTER → \n
  - KEYCODE_DEL → 0x7F (DEL — what mksh/zsh/bash default termios expects for backspace)
  - KEYCODE_FORWARD_DEL → ESC[3~
  - KEYCODE_TAB → \t / KEYCODE_ESCAPE → \x1B
  - DPAD_UP/DOWN/LEFT/RIGHT → ESC[A/B/C/D
  - Ctrl+letter → 0x01..0x1F
  - deleteSurroundingText(N) → emit N × 0x7F

Subtle: BaseInputConnection.commitText synthesizes KeyEvent.ACTION_DOWN+UP via
KeyCharacterMap.VIRTUAL_KEYBOARD for each printable char committed; those route
back through sendKeyEvent. Detect synthesized events (deviceId == VIRTUAL_KEYBOARD)
and skip the byte-write in sendKeyEvent — commitText already wrote. Hardware
keyboards (deviceId >= 0) keep full sendKeyEvent byte mapping.

Regression gate: tools/scripts/test-ime-input-pty.sh asserts commit_count ==
write_count (1:1, no double-write).

## Iteration 20 — Termux smooth + Warp UX scaffold (2026-05-02, commits 3bb475a + 7b0e145)

User mandate: "我就是要在 Android 的 warp 並且終端機功能跟 termux 一樣絲滑".
Two halves shipped together:

(a) **Termux smooth via nativeLibraryDir refactor** — every $PREFIX/bin/* ELF
binary is now packaged as APK lib/<abi>/lib<sanitized>.so. Android's SELinux
domain `untrusted_app` has neverallow on `app_data_file:file execute` since
API 29 (the cause of "zsh dies in 10 ms" through M1–M6) but DOES have execute
on `apk_data_file` which is the label of files in ${nativeLibraryDir}.
Modern Termux uses the same trick. Build pipeline added to android/app/
build.gradle: `extractTermuxBinariesAsLibs{Debug,Release}` opens the
bootstrap zip, ELF-detects each `bin/<name>` entry, sanitizes the name,
writes lib<name>.so to a per-variant intermediate, emits a JSON manifest
into APK assets. Device-side first-launch + every-spawn:
WarpTerminalService.installTermuxBinSymlinks() reads the manifest and
replaces each $PREFIX/bin/<name> regular file (extracted by
bootstrap_install) with a symlink to ${nativeLibraryDir}/lib<name>.so.
Idempotent. 158 ELF binaries shipped (apt, bash, curl, dpkg, find, gawk,
grep, less, sed, tar, xz, zstd, zsh, …); shell scripts in $PREFIX/bin
remain as the original regular files. APK 134 MB → 140 MB. Verified:
zsh prompt renders, `awk --version` = GNU 5.3.2, `bash --version` = GNU
5.3.9, `curl --version` = full HTTP/3 stack with ngtcp2 + nghttp3.

(b) **Warp UX scaffold** — Compose Material 3 chrome around the existing
warpui Vulkan terminal pane, matching Warp Desktop's reference shape:
  - Top app bar: hamburger / "Search sessions, agents, files…" pill /
    new tab + / settings ⚙
  - ModalNavigationDrawer with "Search tabs…" filter + tabs list
  - Bottom prompt composer ("Warp anything…") with cwd-folder icon +
    model picker chip ("● auto (cost-efficient)") + mic + send
  - Material You dynamic color on Android 12+

Compose hosts the existing FrameLayout (SurfaceView + WarpInputView +
AccessoryRow) via AndroidView. Driver-mode escape hatch
`--ez legacy_layout true` skips Compose for backwards-compatible
device-test geometry. New file: dev/warp/mobile/ui/WarpAppTheme.kt +
WarpScaffold.kt.

Lessons:

- **Test "tap the icon and type" every milestone**. Iteration 18+19+20
  surfaced four user-facing bugs that every prior milestone-close had
  signed off on (magenta launcher, hardcoded grid 80×24, Gboard
  silent-input, $PREFIX/bin all blocked). All four traced to "device-
  verified" tests using `am broadcast` to bypass the InputConnection +
  PATH path real users take. Add a "tap the icon and type" gate to
  every milestone close.
- **SELinux > mount flags for app sandbox understanding**. The /data/
  user/0 mount on Galaxy S24 Ultra has `nosuid,nodev` only — no
  `noexec`. So mount-flag-level analysis suggested $PREFIX/bin/* should
  exec. SELinux `neverallow untrusted_app app_data_file:file execute`
  is the actual gate. ${nativeLibraryDir}'s `apk_data_file` label
  bypasses it. Always check `ls -laZ` (SELinux context column) when
  EACCES on execve in app-private paths.
- **The dynamic linker is not bound by the same SELinux rule as
  execve(2)**. zsh's dynamic deps in $PREFIX/lib/* (libpcre2-8,
  libncursesw, libandroid-support, …) are all `app_data_file`-labelled
  but PROT_EXEC mmap on them succeeds — the policy denies execve(2)
  on `app_data_file:file` but permits `execmod` for already-running
  processes. So we ONLY need to relocate the bin/ tree, not the lib/
  tree. The original v1.1 plan estimated 7+ days; the actual refactor
  was a single session.

## Iteration 21 — OSC 133 leak silenced (2026-05-02, commit 2e4453c)

Once iteration-20's Termux refactor put zsh in a real PTY, the rendered
terminal showed `% 133;Alocalhost% 133;B` instead of `localhost%`. The
streaming ANSI parser had no OSC variant: ESC `]` fell into the catch-all
`0x40..=0x5F => Ground` arm which ate the `]` and let the OSC body type
as visible glyphs. Fix: AnsiState::OscString + OscFinish7Bit, ESC `]` →
OscString, consume bytes silently until ST (ESC \\, 0x9c, BEL 0x07).
Buffer cap 4 KB. Two regression tests added:
osc_133_and_title_are_consumed_silently +
osc_8_hyperlink_envelope_silent_text_visible.

zsh prompt now renders cleanly. Affected sequences also fixed in passing:
OSC 0/2 (title), OSC 7 (cwd tracking), OSC 8 (clickable hyperlinks,
emitted by GNU ls --color=auto), OSC 9/777 (notifications).

Lesson: every Warp-style integration we add to the streaming parser
needs paired regression tests for the negative case ("the envelope
bytes do NOT show up in the rendered grid") because the failure mode
is "looks weird" not "crashes". The two tests added here would have
caught iteration-20's leak immediately in CI.
