Each fix exposed another precision gap. Worth it — the round-9 ACs are now F-Droid-deployment-grade. Pattern lesson: when the artifact is sound but ACs need more precision, allow 5-9 rounds of progressive refinement.

M4-S05's atomic extraction can mmap the asset directly only if it's Stored. Found this during M4-S04 codex review.
- **Cross-task closure resolution in Gradle**: `def closure = { ... }` at script level scopes to enclosing block; calling from inside `task { doLast { closure(...) } }` may fail with "no signature" if Gradle's project.ext binding isn't used correctly. Workaround: inline the closure body in the task that needs it. Tradeoff: code duplication vs dependency management.

### Lessons learned (iteration 14)

- **Plan Amendment 6 pattern reinforced**: same Pre-mortem-trip → architecture-pivot pattern as Amendment 5 (M3-S03 cfg-gate→extraction). When the underlying assumption (docker source-compile would Just Work on free CI) breaks fundamentally — not just at one point but as a class of error across 3 attempts — write the amendment, switch to the "obvious other path" (upstream-prebuilts + retarget at our seam), preserve milestone scope.
- **Batched final-stretch closure works**: codex round-by-round on S03/S04/S05/S06/S12 was the slow-careful work needed for risky changes (RUNPATH retargeting, JNI atomic extraction, zsh runtime-config). The remaining S07/S08/S09/S10/S11/S13/S15 were FAR less risky (apt config text override + reproducibility script tweaks + F-Droid YAML + test drivers + close-out doc); batching them through device test-driver-as-verifier instead of round-trip codex saves 3-5 cycles per story without quality loss.
- **Test driver as acceptance evidence**: test-bootstrap-install.sh + test-pkg-install.sh codify the manual device tests I'd been running ad-hoc. Future devs (or Codex for M5) can re-run these scripts to verify regressions. Pattern: every M-S## story whose AC text says "verify on device" should ship a device test script that exercises that AC.
- **Sandbox vs runtime distinction in test design**: `adb shell run-as <pkg>` doesn't have DNS resolver access (that's a feature of the Android sandbox model, not a bug we can fix). Tests that need network — like full `pkg install` end-to-end — must run inside the actual app's PTY context. The driver scripts work around this by validating CONFIG-LEVEL correctness (apt-config dump, zshenv content) without exercising the network path; the actual install is manual M5 territory.
- **Cosmetic vs functional divergence in apt-config**: `Dir::Bin::solvers::` and `planners::` are LIST-APPEND options where the SCALAR overrides only set one entry; the compile-time list defaults survive. apt uses scalar first, falls back to list — so the cosmetic com.termux entries never get hit at runtime. Tolerated by M4-S11 driver; cleanup is M5 polish.
- **Reproducibility is mostly mtime + sort**: byte-identical zip output across two builds at the same upstream snapshot took 3 fixes — sort SYMLINKS.txt entries, sort zip entry order via `find | sort | zip -@`, normalize all mtimes to SOURCE_DATE_EPOCH via Python os.utime. Once these three are in place, two builds produce identical sha256.
- **Recursion guards in zsh sourcing**: zsh_body.sh internally sources `${ZDOTDIR}/.zshenv`, which then sources zsh_body.sh, infinite loop. Codex caught this in M4-S06 round-2 via a real local repro. Fix: `WARP_ZSH_BODY_SOURCING` in-progress sentinel, set BEFORE source, unset AFTER. Process-local (not exported) so each shell session gets its own. Pattern lesson: when sourcing user-extensible scripts, ALWAYS guard against re-entry — don't trust the script to set its own done-flag before it's done.

### Lessons learned (iteration 15)

- **State machine first, UI integration second**: M5-S01 + M5-S03 chose to ship the lock-free atomic state machines + host tests in round-1, deferring touch/Vulkan integration to v1-release. - **Sticky modifier UX is universal**: Ctrl/Alt buttons that toggle for the next keystroke (single-shot) match Termux/Termius behavior and avoid awkward "hold modifier" gestures on a touch screen. Visual feedback (highlighted background when pending) is non-negotiable for discoverability.
- **Chunked clipboard paste is a real fix not a workaround**: Linux PTY canonical-mode line buffer is 4 KB; a single 10K-byte ptyWrite silently drops past that. 4 KB chunks with 1 ms gaps gives the child read() loop time to drain. This is a known-good pattern from terminal apps; we just have to actually do it.
- **`#clear` doesn't work on apt compile-time defaults**: apt.conf is a runtime-config OVERLAY on top of compile-time defaults baked into the binary. `#clear KEY::` clears USER-CONFIG list entries but not the compile-time seeds. Tried and verified on device; deferred to v1-release Option C from-source recompile.

Marking these PARTIAL with explicit "v1-release scope" in verifiedBy is more useful than either claiming PASS (false) or marking incomplete (blocks story counting).
- **Cross-repo commits are clean if main repo's gitignore is set right**: warp-src is gitignored in the main repo (correct — it's a separate git repo). My `git add warp-src/...` from the main repo failed loud-and-clear ("paths are ignored"). Then `git -C warp-src add ...` + `git -C warp-src commit` + `git -C warp-src push` cleanly commits to warp-src's own remote. Pattern: separate add/commit/push for each repo.

### Lessons learned (iteration 16)

- **Self-review pre-commit catches race conditions unit tests can't**: Three different self-review
  passes (code-reviewer + security-reviewer + architect) caught 1 CRITICAL TOCTOU UAF on
  `activeStreamHandle` (commit `f5d716a`'s round-3 review), 2 HIGH races on `current` field
  state-mutation in M6-CO1 (commit `dcce36f`), and 1 hidden self-cancel cascade that broke the
  ENTIRE M6-CO1 feature in production but was masked by the test API key always returning 401
  in ~600 ms (commit `c1dc07a`). Unit tests don't reproduce these without adversarial
  multithreading harnesses; multi-agent reviews caught them by inspection. Worth keeping in the
  M7+ playbook.

- **Diagnostic instrumentation > attempted fix when upstream-blocked**: M4-S14 emoji color
  rendering looked like a bug-to-fix until per-glyph swash content logging (commit `452e3c5`)
  proved swash 0.1.x simply doesn't decode COLR v1 (the modern Noto Color Emoji format on
  Android 15). The fix is upstream (cosmic-text fork → swash 0.2). Documenting 4 alternative
  paths in the result JSON saves any future contributor from repeating the investigation. The
  log lines are also kept in for production diagnostics behind log level Debug.

- **`debounceJob?.cancel()` inside `cancelActiveStream()` was a self-cancel trap**: Pre-existing
  bug in M6-CO1 round-1: `fireSuggestion` defensively called `cancelActiveStream` at its top to
  release any leftover stream, but `cancelActiveStream` ends with `debounceJob?.cancel()` —
  cancelling THE VERY COROUTINE running `fireSuggestion`. The CancellationException fired at
  the next `delay()` in the poll loop, the finally block freed the handle, and the stream was
  released within ~1 ms of starting. With a valid API key this would NEVER produce :DONE:.
  The 401 from the test key arrived in ~600 ms (after the orphan tokio task) — which is what
  let me ship round-1 thinking it worked. Lesson: when adding a "defensive" call that touches
  shared state, trace the side effects all the way down.

- **API consumer surface boundaries matter for v1-prep**: The `Block.output: Vec<u8>` capture
  could have been a stringified field, but storing raw bytes lets us bound the cap precisely
  (64 KB) without worrying about partial multi-byte UTF-8 splitting. JSON dump does
  `String::from_utf8_lossy` at serialization time. Pattern: capture raw, serialize lossy at
  the boundary.

- **AGP signingConfig graceful degradation**: The build.gradle pattern
  `if (hasKeystore) { signingConfig signingConfigs.release }` lets the same source tree build
  signed APKs locally (with `keystore.properties`) AND unsigned APKs in CI / F-Droid (without).
  No conditional compilation, no fork. The keystore.properties file is gitignored; the
  generation instructions live as comments in the same build.gradle block where the config
  reads from it. This is the "secret-where-it's-used" pattern — anyone touching signing
  reads the docs by being in the right file.
