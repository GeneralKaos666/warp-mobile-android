# v1.1 Plan — SELinux-friendly Termux runtime via `nativeLibraryDir`

**Status**: Draft (2026-05-02). Trigger: v1.0 ships with mksh-only shell because every binary in `$PREFIX/bin/` lives under `/data/user/0/dev.warp.mobile/files/usr/bin/` (`app_data_file:s0`-labelled), and Android since API 29 enforces `neverallow untrusted_app app_data_file:file execute` via SELinux. The app's foreground-service runs in `untrusted_app:s0` domain, so `execve` on `$PREFIX/bin/zsh` returns EACCES (errno 13). Real users tap the icon and the launcher-default fallback to `/system/bin/sh` (mksh) is what they get. v1.1 fixes this by relocating the entire Termux runtime into `nativeLibraryDir`.

## §1. Why `nativeLibraryDir` works

Android's package manager extracts JNI libraries (`lib/<abi>/lib*.so` inside the APK) into `/data/app/~~<hash>/<pkg>-<id>/lib/<abi>/`. That directory is labelled `system_lib_file:s0` in SELinux policy, and `untrusted_app` domain is allowed `execute` on `system_lib_file`. Every Android app that ships native libraries already exec's binaries from there (the linker, all `.so` constructors, etc.) so the rule is universal.

The constraint: filenames inside `lib/<abi>/` must start with `lib` and end with `.so`. The package manager strips all other files. So `zsh` would ship as `lib/arm64-v8a/libzsh.so`, `coreutils` as `lib/arm64-v8a/libcoreutils.so`, etc.

This is the same trick Termux uses on modern Android (post-targetSdkVersion-28-deprecation), and what apps like UserLAnd / Termius / shells-as-libraries all use.

## §2. Scope

The bootstrap zip already contains a normal `usr/` tree with `usr/bin/`, `usr/lib/`, `usr/share/`, etc. v1.1 keeps that tree on disk for the read-only assets (terminfo, locale data, etc.) but moves every **executable** out of `usr/bin/` and into `nativeLibraryDir` with the `lib*.so` rename.

Approximate counts to plan against:
- `usr/bin/`: ~80 binaries (zsh, busybox-style coreutils, apt, dpkg, plus libcoreutils tools and tiny helpers).
- Symlinks inside `usr/bin/` (e.g. `[`, `bash` → `mksh`, `awk` → `gawk`): need to be resolved to the canonical filename and re-symlinked at extract time, since `nativeLibraryDir` is read-only.
- Shared libraries already in `usr/lib/lib*.so`: those mostly pass through — they were going to be in `usr/lib/` anyway, but for v1.1 we co-locate the executable-rename'd copies in `nativeLibraryDir`.

## §3. Implementation sketch

### §3.1 Build pipeline (host side)

1. After `build-bootstrap.sh` produces the standard zip, a new step iterates `usr/bin/` and:
   - For each regular file `usr/bin/<name>`: rename to `lib<name>.so`, copy into `_bootstrap-out/lib/arm64-v8a/`.
   - For each symlink `usr/bin/<name>` → `<target>`: record `(name, target)` pair into `_bootstrap-out/bin-symlinks.json`.
2. The bootstrap zip ships **only** the read-only data (`usr/lib`, `usr/share`, `usr/etc`, etc.) plus the symlinks manifest. The renamed binaries are bundled into the APK as JNI libraries.
3. AGP packages `_bootstrap-out/lib/arm64-v8a/lib*.so` into the APK alongside the existing `libwarp_mobile_android_host.so`.

### §3.2 First-launch extraction (device side)

1. `bootstrap_install` keeps doing the existing job for the `usr/lib`, `usr/share`, etc. tree.
2. New step: for each `(name, target)` in `bin-symlinks.json`, create a symlink `$PREFIX/bin/<name>` → `${nativeLibraryDir}/lib<target>.so`.
3. PATH is updated so `${nativeLibraryDir}` (or the symlink-bridged `$PREFIX/bin`) is searched first.
4. Idempotent on subsequent launches: re-checks the symlink targets against `nativeLibraryDir` (which can change `~~<hash>` segment after app update).

### §3.3 PTY shell selection (device side)

`WarpTerminalService.handleSpawn` already prefers `$PREFIX/bin/zsh` if present, falls back to `/system/bin/sh`. Once §3.2 is done, the bridged `$PREFIX/bin/zsh` resolves to `${nativeLibraryDir}/libzsh.so` which IS exec-allowed. The fallback in `startReadLoop` becomes a defense-in-depth net rather than the actual UX path users see.

## §4. Open questions

1. **Apt internals**: does `apt-get install foo` write to `$PREFIX/bin/foo`? If so, the user expects to be able to install new packages and have them work. Either:
   - We need a `nativeLibraryDir`-aware loader that intercepts apt's post-install `chmod +x` and instead emits a "binary installed but read-only — restart app to use" notice, **or**
   - We accept that `apt install` doesn't add new exec-able binaries on Android post-API-29. Termux modern documents this same limitation.
2. **Multi-ABI**: AGP-3+'s split-APK approach already ships per-ABI APKs; the rename pipeline runs inside CI for arm64-v8a only at first, with armeabi-v7a as v1.2 stretch.
3. **Symlink resolution at install-time**: `nativeLibraryDir` is created by the package manager and is read-only for the app, so the symlinks live in `$PREFIX/bin/` (`app_data_file`) pointing INTO `nativeLibraryDir`. Symlinks themselves are exec-able if the target is, which we've verified for `system_lib_file`.
4. **APK size**: ~80 binaries × average ~600 KB = ~50 MB more in the APK (currently 7.4 MB release). Brings the APK to ~60 MB. Still under the F-Droid 100 MB single-APK ceiling, comfortably within the GitHub Releases limit.

## §5. Test plan

1. Build APK with the v1.1 pipeline on arm64-v8a.
2. Install on Galaxy S24 Ultra.
3. `adb shell` → `run-as dev.warp.mobile ls -la lib/arm64/libzsh.so` → confirm presence.
4. `adb shell` → cold-boot via plain launcher Intent → expect zsh prompt rendering on screen (not mksh). Verify with screenshot.
5. Type `ls -la /` and `uname -a` and `whoami` — every command from `$PREFIX/bin/*` should now produce normal output, not `Permission denied`.
6. Re-run M3-S07 block-model driver test with zsh as the auto-spawned shell; confirm Block aggregator picks up Bootstrapped/Precmd/CommandFinished hooks (the hooks need zsh, so this is the reverse direction of the v1.0 device-test where M3-S07 used pre-injected DCS bytes via `TERM_INJECT_RAW`).
7. Verify `apt list --installed` runs (no exec failure on `apt`).
8. Confirm bootstrap-zip reproducibility re-verification still passes (M4-S08 byte-identical re-extract gate).

## §6. Migration notes

- Existing users on v1.0 → v1.1 upgrade: first launch on v1.1 detects the new `bin-symlinks.json` format, replaces the `app_data_file`-labelled binaries in `usr/bin/` with symlinks pointing into the new `nativeLibraryDir`, no user action needed.
- Downgrade path is one-way: v1.1 → v1.0 would need a manual `Clear Data` because v1.0's bootstrap install would skip-over an existing `usr/` that has symlinks instead of binaries.
- F-Droid build needs to publish the same APK structure (lib/ inclusion is automatic; the bootstrap zip just gets smaller).

## §8. Why this isn't blocking v1.0

The v1.0 ship gate is "user can tap the icon and use a working terminal". The mksh fallback meets that bar. Block-model rendering, AccessoryRow, AI ghost-strip, BYOK Settings — all the differentiator features — work on top of mksh because they don't depend on $PREFIX binaries directly; they depend on the PTY pipeline + the renderer + the JNI bridge, none of which are SELinux-blocked.

What v1.0 users LOSE without this fix:
- `ls`, `cat`, `head`, `tail`, `grep`, `find`, `sed`, `awk`, every other GNU coreutils command.
- `apt install` (cannot exec freshly installed binaries).
- zsh's interactive line-editor + the M3-S07 Block-model auto-segmentation (because that depends on zsh's DCS hooks firing during interactive use).

These are real losses, but they're "preview-quality v1.0 with documented limitations" rather than "app doesn't launch" — which is what v1.0 looked like before iteration 18.
